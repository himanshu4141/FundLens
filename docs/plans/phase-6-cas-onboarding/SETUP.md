# Phase 6 — Manual Setup & Testing Runbook

This is the operator runbook for taking Phase 6 from "PRs landed" to "I can test it on dev preview, and prod is ready when the beta opens." It covers **both dev and prod in one pass** so you don't have to do this twice on different days.

> **Strategy recap.** One Resend account, one verified parent domain (`foliolens.in`), two Supabase projects. Inbound addresses are namespaced by subdomain: `cas+<token>@dev.foliolens.in` for dev/preview, `cas+<token>@foliolens.in` for prod. Two separate Resend Inbound Routes, two separate signing secrets, two MX record sets. A leaked dev secret cannot sign a prod webhook. The `dev.` prefix is also user-visible — testers know at a glance which env they're poking at.

## Section 1 — Test M1 (the wizard, no backend setup needed)

Once CI on PR #92 is green, the wizard is ready to test independently of M2 backend. Skip this section if you only want to set up M2 right now.

### a. Web (Vercel `foliolens-dev`)

- Open the Vercel preview URL from the PR comment (`foliolens-dev-…vercel.app`).
- Sign in. If your `user_profile.pan` is already saved, you'll land on the home tabs instead of the wizard. To test the first-run flow, clear it via dev SQL:

  ```sql
  -- DEV Supabase SQL Editor
  update user_profile
     set pan = null, dob = null, kfintech_email = null
   where user_id = '<your-uuid>';

  delete from cas_import where user_id = '<your-uuid>';
  ```

- Drag the browser between mobile (`<1024 px`) and desktop (`≥1024 px`) widths. Mobile shows a full-width column; desktop centres the wizard inside the sidebar shell.

## Section 2 — M2 backend setup runbook (dev + prod, one pass)

The runbook is ordered so DNS propagates while you do the dashboard work. Total wall-clock time: ~30 minutes including DNS waits.

### Step 1 — Apply the migration to both Supabase projects

Both projects need the `cas_inbox_token` column + the `gen_cas_inbox_token()` function + the BEFORE INSERT trigger + the per-row backfill. The migration is `supabase/migrations/20260504020000_user_profile_cas_inbox_token.sql` from PR #93.

**Dev** (auto-deploys on `main` push to `supabase/**` paths via `supabase-deploy-dev.yml`):

- After PR #93 merges into `main`, the workflow runs automatically. No action needed.
- If you want to apply it sooner from the PR branch: GitHub → Actions → **Supabase Deploy (dev)** → Run workflow → branch `feat/cas-resend-inbound-m2`.

**Prod** (manual, behind `workflow_dispatch`):

- After PR #93 merges into `main`, GitHub → Actions → **Supabase Deploy (prod)** → Run workflow → on `main`.
- This is the right time to take a prod backup first. Supabase Dashboard → `<prod-project-ref>` → Database → Backups → Download a recent point-in-time snapshot before triggering.

**Verify both** (Supabase Dashboard → SQL Editor for each project):

```sql
-- Should show one row per user with a unique 8-char token
select user_id, cas_inbox_token from user_profile order by created_at;

-- Sanity: every token unique
select count(*) total,
       count(distinct cas_inbox_token) unique_tokens
  from user_profile
 where cas_inbox_token is not null;
-- expect total == unique_tokens
```

### Step 2 — Add MX records on Cloudflare for both subdomains

Cloudflare Dashboard → `foliolens.in` → DNS → Records.

Resend will give you the exact MX targets in their inbound onboarding modal — they look like `feedback-smtp.<region>.amazonses.com`. Use **whatever Resend tells you to use** for both, with these hosts:

| Type | Name (host) | Priority | Target | TTL |
|---|---|---|---|---|
| MX | `dev` | 10 | _(from Resend)_ | Auto |
| MX | `@` (apex) | 10 | _(from Resend)_ | Auto |

> ⚠️ **Watch for existing apex MX records.** If `foliolens.in` already has Cloudflare Email Routing or any other MX records on the apex, adding a second one creates ambiguity. Either remove the existing apex MX (if it's unused) or run prod inbound on a different subdomain (`mail.foliolens.in`, etc.) and update `<prod-inbound-domain>` accordingly. **The apex is currently used for Resend outbound (magic-link sign-in)** — Resend outbound only needs SPF/DKIM (TXT records), not MX. Safe to add the inbound MX on the apex.

Wait for propagation before continuing:

```bash
dig MX dev.foliolens.in +short
dig MX foliolens.in +short
# Both should return Resend's MX hosts within ~2 minutes.
```

### Step 4 — Set Edge runtime secrets on both Supabase projects

Each project needs **two** secrets: the Resend signing secret + the inbound domain it's responsible for.

**Dev** (Supabase Dashboard → `<dev-project-ref>` → Edge Functions → Manage secrets):

| Key | Value |
|---|---|
| `RESEND_INBOUND_SECRET` | `<dev-resend-secret>` |
| `INBOUND_DOMAIN` | `dev.foliolens.in` |

**Prod** (Supabase Dashboard → `<prod-project-ref>` → Edge Functions → Manage secrets):

| Key | Value |
|---|---|
| `RESEND_INBOUND_SECRET` | `<prod-resend-secret>` |
| `INBOUND_DOMAIN` | `foliolens.in` |

Verify locally if you have the Supabase CLI linked:

```bash
supabase secrets list --project-ref imkgazlrxtlhkfptkzjc
supabase secrets list --project-ref ohcaaioabjvzewfysqgh
# Both should list RESEND_INBOUND_SECRET and INBOUND_DOMAIN.
# Values are not shown — only key names.
```

> **Why also set `INBOUND_DOMAIN` even though prod's value matches the code default?** Two reasons: (1) explicit > implicit — a future env-var typo on prod won't silently fall back to a stale default, and (2) symmetry with dev makes the prod secret list easier to audit.

The client-side address builder reads this env var to decide which subdomain to render in the user's "Auto-refresh inbox" card and Settings row. Match each app environment to its Supabase project.

### Step 6 — Confirm webhook is deployed with `--no-verify-jwt`

Resend cannot send a Supabase JWT; without `--no-verify-jwt`, every inbound webhook is rejected with 401 before reaching the function. Both Supabase deploy workflows already pass this flag for `cas-webhook-resend`, but verify once.

For each project, Supabase Dashboard → Edge Functions → `cas-webhook-resend` → Details. Look for:

- **JWT Verification**: `Off` (or "Skip JWT verification": ✓).
- **Last deployed**: a recent timestamp matching when the workflow ran.

If JWT verification is still on, redeploy explicitly:

```bash
supabase functions deploy cas-webhook-resend --no-verify-jwt --project-ref imkgazlrxtlhkfptkzjc
supabase functions deploy cas-webhook-resend --no-verify-jwt --project-ref ohcaaioabjvzewfysqgh
```

## Section 3 — Validation tests for both environments

Run these after Section 2. The dev tests double as a rehearsal; if dev passes and prod doesn't, the difference is almost always Step 4 (wrong secret on prod) or Step 5 (wrong env var on the prod app build).

### Test 1 — Smoke (function-level, no real user needed)

Send an email **with a small PDF attached** to `cas+ZZZZZZZZ@<env-domain>` from your personal address (Gmail / iCloud — whatever you have).

- **Dev**: send to `cas+ZZZZZZZZ@dev.foliolens.in`. Token alphabet is `[A-HJKMNP-Z2-9]`, so any 8 chars from that set work as a placeholder.
- **Prod**: send to `cas+ZZZZZZZZ@foliolens.in`.

Watch:

- Resend Dashboard → Inbound → Logs. Both routes should show 200 within ~5 sec.
- Supabase Dashboard → Edge Functions → `cas-webhook-resend` → Logs. Expect `[cas-webhook-resend]` lines for **invocation**, **signature OK**, **user resolution failed (404)**.
- The 404 is the correct outcome — no user has `cas_inbox_token = 'ZZZZZZZZ'`. The smoke proves: DNS routes mail to Resend, Resend signs + posts to the right webhook, the function verifies the signature, and the function reaches the lookup step.

If a smoke fails, do not proceed to test 2 — see Troubleshooting below.

### Test 2 — End-to-end (real token, real PDF)

Find your token on each environment's user_profile:

```sql
-- DEV
select cas_inbox_token from user_profile where user_id = '<your-dev-uuid>';

-- PROD
select cas_inbox_token from user_profile where user_id = '<your-prod-uuid>';
```

Forward (or send) a real CAMS / KFintech CAS PDF to `cas+<your-token>@<env-domain>`. Repeat for each env.

Watch the same logs as test 1 — this time user resolution succeeds, parser fires, `import-cas` runs.

Verify import row:

```sql
select id, status, import_source, created_at
  from cas_import
 where user_id = '<your-uuid>'
 order by created_at desc
 limit 1;
-- expect import_source = 'email' and a recent timestamp
```

Open the app on the matching env and confirm the Portfolio reflects the imported funds and transactions.

### Test 3 — Negative checks (security boundary)

For each env:

1. **Tampered signature**: in Resend Dashboard, replay the previous webhook with a corrupted `Svix-Signature` header. Expect 401 from the Edge Function and **no `cas_import` row inserted**.
2. **Unknown token**: send a real PDF to `cas+NOTAREAL@<env-domain>` (alphabet-valid placeholder). Expect 404 in function logs, no row inserted.
3. **Cross-env leak**: send to `cas+<your-prod-token>@dev.foliolens.in` (i.e. a prod token routed to dev). Expect dev function to reject — its `INBOUND_DOMAIN` is `dev.foliolens.in`, so the parser ignores any `to` header that doesn't match. The mail still reaches dev's Resend route (because the address ends in `dev.foliolens.in`), but the function returns 404 because no dev user has that token. **Critically: no prod data is touched.**
4. **Non-PDF attachment**: forward a random email with a `.txt` attachment. Expect the function to log "no PDF attachment" and return 200 silently (no retry, no row).

## Section 5 — Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Resend logs show 0 inbound matches when you send the smoke email | DNS not propagated yet, or MX records pointed at the wrong host | `dig MX <env-domain> +short` — should return Resend's MX targets. If empty, wait 2 min; if wrong, fix in Cloudflare. |
| Resend logs the webhook attempt but Supabase function never gets invoked | Webhook URL typo on the Resend route, or Supabase function not deployed | Compare URL in Resend route vs Supabase Dashboard → Edge Functions → `cas-webhook-resend` → URL. |
| Function logs show 401 on every request | `RESEND_INBOUND_SECRET` mismatch | Resend Dashboard → Inbound → your route → Reveal secret → re-paste into Supabase Edge Functions → Manage secrets. Redeploy not required (secrets reload). |
| Function logs show "INBOUND_DOMAIN not set" or matches no `to` header | `INBOUND_DOMAIN` missing on Supabase Edge runtime | Add it under Edge Functions → Manage secrets and **redeploy** the function (env changes need a redeploy on Supabase Edge). |
| Function returns 404 for a real user that should exist | Token typo in the email's `To` address, or the user's `cas_inbox_token` is null (unbackfilled) | Check Resend log for the actual `to` value vs `select cas_inbox_token from user_profile where user_id = …`. |
| App shows the prod address (`foliolens.in`) on the dev preview build | `EXPO_PUBLIC_INBOUND_DOMAIN` not set on the EAS / Vercel env | Set it for `preview-pr` + `preview-main` (EAS) and `foliolens-dev` (Vercel), then republish OTA + redeploy preview. |
| App shows `dev.foliolens.in` on prod | `EXPO_PUBLIC_INBOUND_DOMAIN=dev.foliolens.in` accidentally set on the production EAS channel or `foliolens` Vercel project | Remove the dev value from those targets and redeploy. The prod default falls back to `foliolens.in`. |
| `cas_import` row inserted but Portfolio empty | Parser returned 0 funds | Function logs include the parser's response body. Common cause: wrong PAN saved on the user's profile. PAN is immutable in-app, so if it's wrong you'll need to update via SQL: `update user_profile set pan = '<correct>' where user_id = '<uuid>'` — only do this for the one row, not in bulk. |
| Webhook fails after merge but worked before | The deploy workflow re-deployed `cas-webhook-resend` without `--no-verify-jwt` | Redeploy explicitly via `supabase functions deploy cas-webhook-resend --no-verify-jwt --project-ref <ref>`. Check the workflow YAML to confirm the flag is still there. |
| Prod migration fails with "schema_migrations drift" | Supabase parity check stale | See `scripts/check_supabase_migration_parity.py` — PR #89 + #90 fixed this; if it returns, run `supabase db push --include-all` from a linked local env to sync. |
